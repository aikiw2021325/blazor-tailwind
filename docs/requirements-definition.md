## 第1章 システム概要

### 1.1 背景と目的

中小企業の法務部門では、契約書のレビュー業務に以下の課題がある。

- 手動レビューの工数が大きく、法務担当者の負荷が高い
- レビュー品質が担当者のスキル・経験に依存し、属人化している
- 過去の契約書やレビュー知見が組織的に蓄積・活用されていない
- 契約書の期限管理やステータス管理が手作業で、漏れが発生しやすい

本システムは、生成AIおよびRAG（Retrieval-Augmented Generation）技術を活用し、契約書のリスク検出・条項分析を支援することで、法務業務の効率化とリスク低減を実現する。

### 1.2 システムスコープ

**対象業務:** 契約書のライフサイクル管理（作成 → レビュー → 承認 → 締結 → 保管 → 更新/終了）

**対象外:**
- 電子署名の法的効力管理
- 会計・ERPシステムとの連携（将来拡張として検討）
- 訴訟管理

### 1.3 前提条件

- オンプレミス環境にデプロイする（契約書データの外部流出防止）
- 利用規模は最大50ユーザー、契約書は年間数百件程度
- 既存の Blazor Server (.NET 10) + Tailwind CSS プロジェクトを基盤として利用する
- AIモデルはローカルLLM（Ollama等）またはVPN経由の外部API接続を想定

## 第3章 機能要件

### 3.1 契約書アップロード・管理

| 機能 | 概要 |
|------|------|
| ファイルアップロード | PDF/Word ファイルのアップロード（ドラッグ&ドロップ対応） |
| OCRテキスト抽出 | スキャンPDF・画像からのテキスト抽出 |

## 第4章 非機能要件

### 4.1 セキュリティ

| 項目 | 要件 |
|------|------|
| 通信暗号化 | HTTPS (TLS 1.2以上) |
| データ暗号化 | 保存データの暗号化（at rest） |
| 認証 | セッションタイムアウト（30分） |
| 認可 | RBAC + 部門スコープによるデータ分離 |
| AIデータ保護 | ローカルLLM利用時はデータが外部に出ない構成。外部API利用時はVPN経由 |
| 契約書アクセス制御 | 契約書単位でのアクセス権設定 |
| IPアドレス制限 | 社内ネットワークからのアクセスのみに制限 |

### 4.2 パフォーマンス

| 項目 | 目標値 |
|------|--------|
| 画面表示応答時間 | 2秒以内（一般画面） |
| AIレビュー処理時間 | 60秒以内（10ページ以下の契約書） |
| ファイルアップロード | 50MB以下のファイルを30秒以内 |
| 同時接続ユーザー数 | 50ユーザー |
| 全文検索応答時間 | 3秒以内 |

### 4.3 可用性・運用

| 項目 | 要件 |
|------|------|
| 稼働時間 | 平日8:00-22:00（計画停止を除く） |
| バックアップ | 日次フルバックアップ、RPO 24時間 |
| ログ管理 | アプリケーションログ・監査ログの保存（1年以上） |
| 監視 | ヘルスチェック、エラー率の監視 |
| データ移行 | 既存契約書のインポート手順を提供 |

### 4.4 法的・コンプライアンス

- 個人情報保護法への準拠
- 電子帳簿保存法への対応（検索性要件）
- 契約書データの法定保存期間の遵守
- AI利用に関する免責事項の明示（AIの出力は法的助言ではない旨の表示）

## 第5章 システムアーキテクチャ概要

### 5.1 全体構成

```
[ブラウザ]
    |
[Blazor Server (.NET 10) + Tailwind CSS]
    |
    +-- [認証] ── ASP.NET Identity
    |
    +-- [ビジネスロジック層]
    |       +-- 契約書管理サービス
    |       +-- ワークフローサービス
    |       +-- 通知サービス
    |       +-- レポートサービス
    |
    +-- [AI/RAG層]
    |       +-- AIレビューサービス (Semantic Kernel)
    |       +-- Embeddingサービス
    |       +-- RAGオーケストレーター
    |
    +-- [データ層]
    |       +-- RDB (PostgreSQL) ── EF Core
    |       +-- ベクトルDB (pgvector)
    |       +-- ファイルストレージ (ローカル / NFS)
    |
    +-- [外部連携]
            +-- LLM (ローカルOllama / VPN経由外部API)
            +-- OCRエンジン
            +-- SMTPメールサーバー
```

### 5.2 技術スタック

| レイヤー | 技術 |
|----------|------|
| フロントエンド | Blazor Server Interactive + Tailwind CSS |
| バックエンド | ASP.NET Core (.NET 10) |
| ORM | Entity Framework Core |
| AIオーケストレーション | Microsoft Semantic Kernel |
| LLM | ローカル: Ollama (Llama系/日本語対応モデル)、外部: Azure OpenAI (VPN経由) |
| Embedding | ローカル: Ollama Embedding、外部: OpenAI text-embedding-3-small |
| ベクトルDB | pgvector (PostgreSQL拡張) |
| RDB | PostgreSQL |
| ファイルストレージ | ローカルファイルシステム / NFS |
| OCR | Tesseract OCR またはオンプレ対応OCRエンジン |
| 認証 | ASP.NET Identity |
| ログ | Serilog |
| コンテナ | Docker / Docker Compose |

### 5.3 デプロイ構成

オンプレミスサーバーにDocker Composeで以下のコンテナを配置する想定。

```
docker-compose.yml
├── blazor-app        # Blazor Server アプリケーション
├── postgresql        # PostgreSQL + pgvector
├── ollama            # ローカルLLM
└── redis             # キャッシュ（必要に応じて）
```

## 第6章 データモデル概要

### 6.1 主要エンティティ

| エンティティ | 説明 | 主要属性 |
|-------------|------|----------|
| Contract | 契約書 | Id, Title, ContractType, Status, PartyA, PartyB, EffectiveDate, ExpirationDate, Amount |
| ContractVersion | 契約書バージョン | Id, ContractId, VersionNumber, FilePath, FileHash |
| ReviewSession | AIレビューセッション | Id, ContractVersionId, ReviewType, Status, StartedAt, CompletedAt |
| ReviewResult | レビュー結果 | Id, ReviewSessionId, ClauseReference, RiskLevel, Finding, Suggestion |
| ReviewComment | 人間レビューコメント | Id, ContractVersionId, UserId, Comment, Status |
| Template | テンプレート | Id, Name, ContractType, Content, Version |
| Workflow | ワークフロー定義 | Id, Name, ContractType, Steps |
| ApprovalRequest | 承認依頼 | Id, ContractId, WorkflowId, CurrentStep, Status |
| ApprovalAction | 承認アクション | Id, ApprovalRequestId, ApproverId, Action, Comment |
| KnowledgeDocument | ナレッジ文書 | Id, Title, DocumentType, Content, EmbeddingStatus |
| KnowledgeChunk | チャンク | Id, KnowledgeDocumentId, ChunkIndex, Content |
| Notification | 通知 | Id, UserId, Type, Title, Message, IsRead |
| AuditLog | 監査ログ | Id, UserId, Action, EntityType, EntityId, Details, Timestamp |
| User | ユーザー | Id, Name, Email, Role, DepartmentId |
| Department | 部門 | Id, Name, ParentDepartmentId |
| Tag | タグ | Id, Name, Category |

### 6.2 設計方針

- Contract を中心としたリレーション設計
- 全エンティティに共通カラム（CreatedAt, UpdatedAt, CreatedBy, UpdatedBy）を持つ BaseEntity を継承
- ソフトデリート（IsDeleted フラグ）を適用
- 多対多関係（Contract-Tag等）は中間テーブルで管理

**目標:** 高度機能・運用機能

- 修正提案、比較分析、メタデータ自動抽出、ベクトル検索

## 付録: リスクと課題

| リスク | 対策 |
|--------|------|
| ローカルLLMの精度 | 日本語対応モデルの選定・評価を事前に実施。外部APIへのフォールバック構成 |
| オンプレミスのGPUリソース | LLM推論に必要なGPU要件の事前調査。CPU推論も選択肢として検討 |
| AI出力の法的リスク | 「AIの出力は参考情報であり法的助言ではない」旨の免責表示を徹底 |
| データ移行 | 既存契約書のフォーマット・ボリュームの事前調査 |
| LLM APIコスト（外部利用時） | 利用量の監視・制限機能の実装 |
